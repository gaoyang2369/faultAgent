# 回答终止功能实现计划（2026-04-17）

## 背景

当前聊天主链路已经具备真实 SSE / 工具调用 / 历史恢复能力，但“停止生成”仍未形成真正工程可用的闭环：

- 前端只有关闭 EventSource 的半套能力，没有明确可点击的停止交互
- 后端缺少显式 cancel registry / stop endpoint
- 中断后部分回答的 UI 恢复和历史表现尚未统一

## 目标

以最小侵入方式补齐“停止回答”能力，覆盖：

1. 前端生成中可点击“停止生成”
2. 前端点击后立即停止接收后续 token，并恢复可输入状态
3. 后端收到 stop 信号后尽快结束当前流式请求，避免继续无意义推送
4. 工具链/长文本场景下尽量缩短中断延迟
5. 部分回答保留且刷新后显示不乱

## 执行步骤

1. 审计现有前端 EventSource、`useChatStream`、输入区和后端 `streaming.py`/`app.py`
2. 增加最小的 stream cancellation registry 与 `/chat/stop`
3. 前端为每次流式请求引入 `stream_id`，停止时并发关闭 SSE + 发送 stop 请求
4. 修复中断后的消息状态与本地缓存/历史恢复合并
5. 启动前后端并做真实回归：短回复、长回复、工具调用中断、停止后再提问、刷新恢复

## 风险与边界

- 上游模型/工具若正在阻塞执行，无法保证 100% 硬中断，只能做到尽快取消等待和结束请求上下文
- 中断后的部分回答优先保证前端缓存恢复和会话一致性，不贸然把半截回答写进 Agent 持久上下文
