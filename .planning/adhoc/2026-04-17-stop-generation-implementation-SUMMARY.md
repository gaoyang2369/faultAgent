# 回答终止功能实现总结（2026-04-17）

## 已完成内容

- 为每次 `/chat/stream` 请求引入 `stream_id`，前端停止时会同时关闭 SSE 并调用 `/chat/stop`
- 后端新增轻量取消注册表，流式主循环、工具前后和本地开发模式都能感知取消信号并尽快结束
- 前端补齐“停止生成”按钮、停止中的状态同步、部分回答保留与中断提示
- 历史恢复层补了“尾部中断 assistant”与本地缓存合并规则，避免刷新后消息错乱
- 任务进度面板在中断后会同步切到“已停止生成”展示态，不再残留“执行中 / 进行中”
- 回归覆盖：长文本中断、工具调用中断、停止后立即再次提问、刷新后恢复、日志可观测性

## 已确认结果

- 点击停止后，前端会立即停止接收后续 token，并恢复可继续输入
- 后端会收到 `/chat/stop`，`streaming.py` 会记录取消日志并提前结束流式请求，不再发送 `complete`
- 长文本与工具调用场景都能真实中断，日志中可见 `tool_start` / `tool_end` / `duration_ms` 与取消信号
- 停止后可立即继续在同一线程提问，不会卡死或污染后续对话

## 当前边界

- 若上游模型或工具正处于阻塞执行中，无法保证 100% 硬中断，只能做到尽快结束等待并停止后续推送
- 部分回答优先通过前端本地缓存恢复；若后端 checkpoint 尚未提交该半截 assistant，则冷恢复时可能只看到用户消息
- 这条边界是有意保守处理，目的是避免把未完成回答直接写入 Agent 持久上下文，影响后续推理一致性
